Features Overview

Everything Uplink can do ‚Äî powered by OpenClaw, controlled by you.

Built on OpenClaw

Uplink is designed as a frontend for OpenClaw ‚Äî the open-source AI agent platform. Every feature in Uplink leverages OpenClaw's capabilities under the hood:

  • Chat & streaming ‚Äî Powered by OpenClaw's gateway API (OpenAI-compatible)
  • Satellites ‚Äî Uplink's multi-provider switching maps directly to OpenClaw's model routing
  • Session persistence ‚Äî Your conversation history lives in OpenClaw's session store, surviving restarts and device switches
  • Context tracking ‚Äî The token counter in the header reads real-time context usage from OpenClaw's session data
  • Gateway proxy ‚Äî Uplink proxies WebSocket connections directly to OpenClaw for real-time streaming

Local-First Architecture

Uplink runs as a Node.js server on localhost:3456. Your conversations, settings, and data live on your machine. The AI connection goes through the OpenClaw gateway running locally, so your prompts and responses never touch third-party servers unless you explicitly configure an external endpoint.

When your internet goes down, Uplink keeps serving. The web UI, settings, and conversation history are all local. Only the AI inference requires a working gateway connection.

Text Chat

The core experience. A clean, responsive chat interface in your browser. Features include:

  • Streaming responses ‚Äî Tokens appear in real-time as the AI generates them
  • Markdown rendering ‚Äî Code blocks, tables, lists, and inline formatting
  • Conversation history ‚Äî All chats are persisted and searchable
  • System prompts ‚Äî Set per-conversation or global personality/instructions
  • Multi-device sync ‚Äî Open the same Uplink instance from any device on your network

Satellites

Satellites are independent AI conversations with isolated context. Each satellite maintains its own message history, system prompt, and context window ‚Äî completely separate from your other conversations. Think of them as parallel chat sessions that don't interfere with each other.

  • Independent AI conversations ‚Äî Each satellite is a separate conversation with its own isolated context and memory
  • Isolated context windows ‚Äî Messages in one satellite don't affect the context or token count of another
  • Create and switch freely ‚Äî Spin up new satellites and jump between them from the satellite switcher in the header
  • Clean slate on demand ‚Äî Starting a new satellite gives you a fresh conversation without losing your other chats
  • Same gateway ‚Äî All satellites route through your single OpenClaw gateway ‚Äî they're separate sessions, not separate AI providers
  • Embeddable ‚Äî Part of OpenClaw's embed and widget system, allowing chat interfaces to be embedded in other pages or applications
‚Ñπ Satellites ‚â† Providers Satellites are session-level separation, not model-level. They all use the same OpenClaw gateway and whatever model it's configured with. They're part of OpenClaw's embed/widget architecture.

End-to-End Encryption

Uplink supports optional AES-256 encryption for all conversations. Enable it on first run or later in Settings ‚Üí Privacy. When enabled:

  • All messages are encrypted at rest on the server
  • Your encryption password never leaves your browser
  • Zero-knowledge design ‚Äî the server cannot read your messages
  • If you forget your password, conversations are unrecoverable (by design)
‚Ñπ How It Works Your password is used to derive an AES-256 key via PBKDF2 in the browser. Messages are encrypted client-side before being sent to the server. The server only ever sees ciphertext.

OpenClaw Gateway

Uplink connects to AI exclusively through the OpenClaw gateway. OpenClaw provides an OpenAI-compatible API that handles model routing, session management, and streaming ‚Äî Uplink is built to take full advantage of these features.

üí° Why OpenClaw? OpenClaw isn't just an API proxy ‚Äî it's an AI agent platform with session memory, multi-model routing (satellites), tool execution, and more. Uplink is built to surface all of this through a clean chat interface. Features like the satellite switcher, context tracker, and persistent conversations all depend on OpenClaw's session architecture.
‚ö† Other endpoints While the API format is OpenAI-compatible, pointing Uplink at a raw OpenAI endpoint, llama.cpp, or LM Studio will give you basic chat only. Features like satellites, session persistence, context tracking, and gateway status won't work without OpenClaw.

Artifacts

The Artifacts panel is a built-in document viewer for agent-generated files ‚Äî PRDs, reports, code, notes, anything your agent creates. Browse, search, and read documents without leaving the chat.

Where Artifacts Live

Uplink checks these locations in order and uses the first one it finds:

  1. ARTIFACTS_DIR environment variable (if set)
  2. artifacts/ in the parent directory of your Uplink install (the OpenClaw workspace)
  3. artifacts/ inside the Uplink directory itself

Session Keys & Multi-Platform Use

Uplink uses session keys to track conversation history. Your primary satellite uses the key agent:main:main ‚Äî the same key the OpenClaw Dashboard uses ‚Äî so conversation history stays in sync between the two.

Other satellites get isolated keys like agent:main:uplink:satellite:<id>, and other messaging platforms (Telegram, Discord, Signal, etc.) use their own key formats.

‚ö† Cross-platform memory This means conversation history is per-platform. If you chat with your agent on Telegram and then switch to Uplink, the agent won't remember what was said on Telegram ‚Äî it's a separate session with separate history.

The Fix: Workspace Memory

Instead of relying on conversation history (per-session), have your agent write important context to workspace files. Files like MEMORY.md are injected into every session automatically, regardless of platform.

Quick approach:

  1. Add a memory rule to your AGENTS.md: "After completing any task, write a summary to memory/YYYY-MM-DD.md. Always check memory files before answering questions about project status."
  2. Before switching platforms, tell your agent "memdump" ‚Äî this saves current context to a file.
  3. When you open the other platform, your agent reads the workspace files and picks up where you left off.
üí° Why it works this way Separate session keys are intentional ‚Äî you wouldn't want Telegram messages interleaved into your Uplink history. Workspace files are the bridge that carries context across all platforms. Conversation history is per-session. Workspace files are global.
